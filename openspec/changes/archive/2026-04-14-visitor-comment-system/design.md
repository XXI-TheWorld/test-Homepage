## Context

现有项目是一个个人主页（LinkPage），采用日式木纹边框设计风格，支持明暗主题切换和背景音乐播放。页面只有展示性链接，缺少访客互动功能。

访客希望能留下反馈或打招呼，但无需登录账户系统。

## Goals / Non-Goals

**Goals:**
- 提供一个独立的评论区页面 `comments.html`
- 展示评论列表，包含头像、昵称、时间、内容
- 支持访客输入昵称和留言提交
- 使用 localStorage 实现本地持久化存储
- 保持与主页一致的视觉风格

**Non-Goals:**
- 不实现后端服务器
- 不实现用户登录/认证
- 不实现评论编辑/删除功能
- 不实现评论分页（数据量小）

## Decisions

### 1. 采用独立 HTML 页面而非模态框
**决定**: 创建独立的 `comments.html` 页面
**原因**: 评论区可能有较多内容，独立页面避免占用主页面空间；与现有页面结构一致（都是跳转页面）
**替代方案**: 考虑过模态框，但会导致页面复杂度和交互冲突

### 2. 使用 localStorage 而非后端 API
**决定**: 评论数据存储在浏览器 localStorage
**原因**: 纯前端实现，无需后端服务，适合个人页面；数据仅保存在用户本地，隐私安全
**替代方案**: 考虑过 Firebase/Supabase 等 BaaS，但增加外部依赖；考虑过 LocalStorage + 云同步，但复杂度高

### 3. 头像使用昵称首字母生成
**决定**: 根据昵称前两个字符生成圆角方块头像，配色基于昵称 hash
**原因**: 无需用户上传，减少复杂度和存储成本；视觉效果与网站风格协调
**替代方案**: 考虑过随机预设头像，但缺乏个性化

### 4. 复用现有样式变量
**决定**: 评论页面使用与 index.html 相同的 CSS 变量和 wood-grain 渐变边框
**原因**: 保持品牌一致性，减少重复代码
**替代方案**: 考虑过独立设计，但会破坏视觉统一性

## Risks / Trade-offs

| 风险 | 影响 |  Mitigation |
|------|------|-------------|
| localStorage 达到 5MB 限制 | 评论数据无法继续保存 | 提示用户清理旧评论或导出数据 |
| 换设备/浏览器数据丢失 | 访客历史评论不可见 | 明确的"本地保存"提示，避免用户误以为云同步 |
| 昵称/内容无过滤 | 可能出现不当内容 | 添加敏感词过滤（基础版） |
